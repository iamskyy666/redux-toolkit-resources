This is a clean, real-world async slice. Now let’s dissect it properly — not just what it does, but how it behaves internally.

We’ll go step by step.

---

# Step 1️⃣: The Thunk

```js
export const fetchUsers = createAsyncThunk("users/fetchUsers", async () => {
  const res = await fetch("https://jsonplaceholder.typicode.com/users");
  return await res.json();
});
```

We are using `createAsyncThunk` from **Redux Toolkit**.

## What this line actually creates

```js
createAsyncThunk("users/fetchUsers", async () => { ... })
```

This does NOT just create one action.

It automatically creates **three action types**:

```
users/fetchUsers/pending
users/fetchUsers/fulfilled
users/fetchUsers/rejected
```

And it creates a thunk function called `fetchUsers`.

---

## What happens when we dispatch it?

```js
dispatch(fetchUsers())
```

Internally:

### 1️⃣ It immediately dispatches:

```
users/fetchUsers/pending
```

### 2️⃣ It runs the async function:

```js
const res = await fetch(...)
return await res.json()
```

If successful:

### 3️⃣ It dispatches:

```
users/fetchUsers/fulfilled
```

with payload = users data

If it fails:

### 4️⃣ It dispatches:

```
users/fetchUsers/rejected
```

with error info

That’s the entire lifecycle.

We didn’t write those actions manually — Redux Toolkit generated them.

---

# Step 2️⃣: The Slice

```js
export const usersSlice = createSlice({
  name: "users",
  initialState: {
    users: [],
    loading: false,
    error: null,
  },
```

This defines a slice of global state.

If we register this reducer in our store like:

```js
reducer: {
  users: usersReducer
}
```

Our global state shape becomes:

```js
{
  users: {
    users: [],
    loading: false,
    error: null
  }
}
```

Yes — it’s nested. That’s normal.

---

# Step 3️⃣: Why No `reducers` Field?

Notice we didn’t define:

```js
reducers: { ... }
```

Because this slice does not define normal synchronous reducers.

Instead, it reacts to external actions generated by the thunk.

That’s why we use `extraReducers`.

---

# Step 4️⃣: Understanding `extraReducers`

```js
extraReducers: (builder) => {
```

We use the builder pattern here.

This lets us listen to actions created outside this slice.

Remember:

`fetchUsers` is created outside the slice.

So we handle its lifecycle actions inside `extraReducers`.

---

# Step 5️⃣: Handling `pending`

```js
.addCase(fetchUsers.pending, (state) => {
  state.loading = true;
});
```

When we dispatch `fetchUsers()`:

The FIRST action triggered is:

```
users/fetchUsers/pending
```

This reducer runs.

We set:

```js
loading = true
```

Now our UI can show a spinner.

Important:

Redux Toolkit uses Immer internally, so this mutation:

```js
state.loading = true
```

is actually converted into an immutable state update safely.

---

# Step 6️⃣: Handling `fulfilled`

```js
.addCase(fetchUsers.fulfilled, (state, action) => {
  state.loading = false;
  state.users = action.payload;
});
```

If the API call succeeds:

* `loading` becomes false
* `action.payload` contains the fetched users array
* We store it in `state.users`

Now the UI can render the users.

---

## What exactly is inside `action` here?

For fulfilled:

```js
{
  type: "users/fetchUsers/fulfilled",
  payload: [ ...users data... ]
}
```

The payload is whatever we returned:

```js
return await res.json();
```

That return becomes the payload.

---

# Step 7️⃣: Handling `rejected`

```js
.addCase(fetchUsers.rejected, (state, action) => {
  state.loading = false;
  state.error = action.error.message;
});
```

If fetch fails:

Redux Toolkit dispatches rejected with:

```js
{
  type: "users/fetchUsers/rejected",
  error: {
    message: "Error message here"
  }
}
```

We:

* Stop loading
* Store error message

Now the UI can show an error state.

# Full Lifecycle Summary

Let’s simulate a real execution.

We call:

```js
dispatch(fetchUsers())
```

### Timeline:

**Step 1**
State:

```js
{ users: [], loading: false, error: null }
```

**Step 2 — pending runs**

```js
{ users: [], loading: true, error: null }
```

**Step 3 — fulfilled runs**

```js
{
  users: [ ...API data... ],
  loading: false,
  error: null
}
```

Or if error:

```js
{
  users: [],
  loading: false,
  error: "Network error"
}
```

Clean and predictable.

# Small Improvement We Could Add

Right now, we don’t reset error when a new request starts.

Better practice:

```js
.addCase(fetchUsers.pending, (state) => {
  state.loading = true;
  state.error = null;
});
```

Otherwise, old error messages might remain.

That’s production-level thinking.
